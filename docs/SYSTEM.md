# Домоуправител — документация на системата

Този документ описва как работи приложението, основните модули и съпоставка с продуктовите изисквания.

**План за следващи стъпки по имплементация:** [`IMPLEMENTATION_PLAN.md`](./IMPLEMENTATION_PLAN.md)

---

## 1. Какво е системата

Уеб приложение (PWA) за управление на етажна собственост: единици (апартаменти, гаражи и др.), приходи, разходи, задължения/плащания, документи, такси (само за администратор).  
**Frontend:** React 18, TypeScript, Vite. **Backend:** Supabase (PostgreSQL, Auth, Storage, Row Level Security).

---

## 2. Конфигурация

| Променлива (`.env`) | Назначение |
|---------------------|------------|
| `VITE_SUPABASE_URL` | HTTPS адрес на проекта: `https://<project-ref>.supabase.co` |
| `VITE_SUPABASE_ANON_KEY` | Публичен ключ за клиента (от **Settings → API**). Достъпът до данни се ограничава от RLS в базата. |

За директно свързване към Postgres (напр. `psql`, restore на backup) се ползва **отделен** connection string от **Database → Connect** (препоръчително Session pooler), не anon ключът.

Схема на базата: `database_v2.sql` (и допълнителни скриптове в root на репото при нужда).

---

## 3. Потребители и роли

- Потребителите се удостоверяват чрез **Supabase Auth** (имейл/парола).
- Ролята за приложението се пази в таблица **`public.users`** (`admin` | `editor` | `viewer`), свързана с `auth.users`.
- При нова регистрация тригерът създава ред в `users` с роля по подразбиране **`viewer`**; при регистрация от UI може да се зададе друга роля според логиката в `AuthContext`.
- **`canEdit()`** в приложението: **admin** и **editor** могат да редактират поддържаните екрани; **viewer** е основно за преглед.

### Меню според роля

- **Viewer:** Начало, Задължения, Приходи, Разходи (без Единици, Документи, Такси).
- **Admin / Editor:** същото + Единици + Документи; **Такси** само за **admin**.

---

## 4. Маршрути и какво правят

| Път | Страница | Назначение |
|-----|----------|------------|
| `/login` | Вход | Автентикация |
| `/` | Начало (табло) | Обобщение: брой единици, суми приходи/разходи, брой документи, **баланс** (приходи − разходи) |
| `/units` | Единици | Управление на единици (тип, номер, площ, собственик и др.); при админ — свързана логика за потребители към единици |
| `/income` | Приходи | Списък и запис на приходи (типове: входна такса, паркоместо, други) |
| `/expenses` | Разходи | Разходи и разпределение по единици (равно / по площ / ръчно) |
| `/obligations` | Задължения | Плащания (`payments`): единица, сума, период от–до, дата на плащане; ръчни записи без ред в `income` (миграция `database_migrations/002_payments_simple.sql`) |
| `/documents` | Документи | Качване в Storage bucket `documents`, метаданни в `documents` |
| `/fees` | Такси | Дефиниране на такси (само admin) |

Данните се зареждат от Supabase при **отваряне на страницата** и след **локални промени** (запис/изтриване). За да се видят промени, направени от **друг** потребител, обикновено е нужно **ново зареждане** на страницата или навигация (няма задължителен Realtime push).

---

## 5. Ключови таблици (логически)

- **`users`** — роли към акаунти.
- **`units`** — единици в блока.
- **`income`**, **`expenses`**, **`expense_distributions`** — финанси.
- **`payments`** — плащания по единици; опционална връзка към `income`; полета `period_start` / `period_end` за регистриран период.
- **`fees`** — конфигурация на такси.
- **`documents`** — файлове (път в Storage + метаданни).

Пълни дефиниции: `database_v2.sql`.

---

## 6. Баланс и „актуални“ данни

- **Баланс** на началния екран: сума от всички приходи минус сума от всички разходи (глобално за системата), преизчислява се при зареждане на таблото.
- **„Актуално“** в смисъл *всеки да вижда верните стойности от базата*: при всяко отваряне/презареждане на екрана се правят нови заявки към Supabase. Няма изискване към live синхронизация между отворени сесии без презареждане.

## 7. Изисквания — съпоставка

Доколкото са формулирани за продукта, статусът е следният.

### 7.1 Общи

| Изискване | Статус | Бележки |
|-----------|--------|---------|
| Вижда се актуалното състояние (при зареждане на екрана) | **Покрива се** | Данни от Supabase при всяко зареждане на страницата. |
| Приходи и разходи | **Покрива се** | Отделни страници + табло с обобщени суми. |
| Филтър по година (финанси) | **Не е имплементирано** | Няма UI филтър по година върху приходи/разходи. |
| Баланс = приходи − разходи, автоматично | **Покрива се** | На **Начало**. |
| Админ качва документи; потребители за разглеждане | **Частично** | Качване за **admin и editor** (`canEdit()`). **Viewer** няма достъп до меню „Документи“. |
| Секция задължения | **Покрива се** | Страница **Задължения** (`payments`). |
| Таблица пример: Апартамент / Дължи / Платено / Остатък | **Частично** | Има списък по **плащания** с единица, сума, статус; **няма** обобщаваща таблица по апартамент с три колони „дължи / платено / остатък“. |
| Модул „Форум“ | **Не е имплементирано** | — |
| Съобщения с история (само админ пише, всички четат) | **Не е имплементирано** | — |
| Меню: Табло, Финанси, Документи, Задължения, Форум, Съобщения, Потребители | **Частично** | Има **Начало**, отделно **Приходи/Разходи** (не един „Финанси“), **Документи**, **Задължения**; **няма** Форум/Съобщения; **няма** страница „Потребители“ — има **Единици** и свързани потребители там. |

### 7.2 Документи

| Изискване | Статус | Бележки |
|-----------|--------|---------|
| Списък с файлове, PDF/снимки | **Покрива се** | Storage + таблица `documents`. |
| Свързване с разход (фактура) | **Частично** | В схемата има `related_type` / `related_id`; UI е ориентиран към стар модел с **`floors`** — при база само с **`units`** може да се наложи подравняване на кода/данните. |

---

## 8. Известни технически бележки

- **Документи и `floors`:** страницата `Documents` очаква таблица `floors`, докато `database_v2.sql` ползва **`units`**. При работа само с v2 схема може да има грешки или празни падащи списъци — за корекция: подравняване на UI към `units` и полета в `documents`.
- **Production build:** при `npm run build` може да има TypeScript предупреждения в отделни страници — проверка преди deploy.
